INSIGHTS
核心观点总结
观点 01

编译 > 检索

传统 RAG 每次从零检索，知识没有积累。LLM Wiki 让 Agent 主动将资料"编译"成结构化页面，知识一旦入库就永久留存并持续进化。

"大部分 token 现在都花在'操纵知识'而不是写代码上了。" — Karpathy
观点 02

三层架构 = 不可变源 + 编译库 + 规则引擎

Raw（原始资料，永不修改）→ Wiki（LLM 自动生成的结构化知识）→ Schema（CODEBUDDY.md，定义编译规则）。三层分离，各司其职。

"Wiki 就像一个'第二大脑'，你很少手动写内容，LLM 负责维护。"
观点 03

知识复利：越用越聪明

每次 Ingest 新资料，Agent 会更新旧页面、发现新关联、建立反向链接。知识达到临界点后，Query 的质量和深度呈指数增长。

"Karpathy 更激进，追求'知识自己长出来'的编译式 Wiki。"
观点 04

三大范式各有所长

编译式（LLM Wiki）= 深度积累
检索式（IMA / RAG）= 便捷查询
创作式（YouMind）= 快速产出
最佳实践：三者结合使用。

观点 05

Harness 是知识层的守护者

LLM Wiki 解决"知识怎么积累"，Harness Engineering 解决"Agent 怎么靠谱执行"。CODEBUDDY.md 本身就是一种 Harness——它约束 Agent 的行为边界。

观点 06

CodeBuddy IDE 是最佳落地工具

Craft 模式原生支持多文件 Agent 编辑，Markdown 处理一流，CODEBUDDY.md 规则加载稳定。配合 Obsidian Graph View，形成"写+看"的完美闭环。

LLM Wiki 一句话

知识怎么越积累越聪明 —— 让 LLM 帮你写一本永远在更新的个人百科全书，重点在"知识自己长大"。

Harness 一句话

Agent 怎么靠谱地干活 —— 给 LLM 套上工程化控制框架（规则、监控、回滚），确保它不乱跑、不翻车。

两者的协同关系

LLM Wiki 提供知识底座（长期记忆 & 结构化知识），Harness 提供行为保障（Agent 可靠执行）。
你的 CODEBUDDY.md 既是 Wiki 的 Schema（规则层），也是 Agent 的 Harness（控制层）——它同时定义了"知识怎么组织"和"Agent 怎么行动"。
