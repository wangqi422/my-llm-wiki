# Skill: Distill（知识蒸馏）

> **触发词**: `distill` / `蒸馏` / `知识蒸馏` / `合并页面`
> **职责**: 定期将 wiki 中的细粒度页面压缩、合并、冻结，保持知识库越转越精炼
> **权威来源**: 本 Skill 是 CODEBUDDY.md §3.4 的标准操作实现——ingest 是第一次压缩，distill 是第二次压缩

---

## 核心理念

> **数据飞轮要越转越精炼，不是越转越膨胀。**
> Growth ≠ Learning。99 页不如 30 页精品。

蒸馏的本质：**抽象和压缩**，不是 1:1 映射。

---

## 何时触发

1. **周度蒸馏**（推荐每周一执行）：回顾本周所有新增/更新的 wiki 页面
2. **主动蒸馏**（任何时候）：当发现概念页过于碎片化时
3. **Insight 冻结**：当某个 insight 页超过 100 行时

---

## 第一步：信息衰减扫描

遍历所有 `wiki/concepts/` 和 `wiki/entities/` 页面，对每个页面判断：

### 1.1 同一信号的重复表述

如果一个事实在多次日报更新中被追加了多个版本（如"Kling有统一架构" → "统一架构得到验证" → "统一架构竞品被迫回应"），**只保留最新、最强的版本**，删除或折叠旧版本。

**规则**：
- 同一信号的多次更新 → 保留**一条最精炼的总结**
- 旧版本移入折叠区 `<details><summary>历史版本</summary>...</details>`
- 每个事实只保留**最强来源**

### 1.2 信息密度检查

每个概念页必须通过**密度测试**：
- 如果这个页面的核心内容可以用 **3 句话** 说清楚 → 它不应该是独立页面，应合并到上级概念
- 如果这个页面的 Related 超过 10 个 → 需要精简到 ≤10

---

## 第二步：概念页合并

### 2.1 识别合并候选

以下情况应合并为一个更高层的概念页：

| 信号 | 动作 |
|------|------|
| 多个页面在说同一件事的不同角度 | 合并为一个页面，用 H2 分节 |
| 页面内容重叠度 >50% | 合并，去重 |
| 页面只有 1 个来源且 <30 行 | 考虑合并到上级概念 |
| 两个页面始终一起被引用 | 合并 |

### 2.2 合并粒度标准

**一个概念页应代表一个"可以讨论 30 分钟的话题"。**

- 如果 5 分钟就能讲完 → 太细，应合并
- 如果需要 2 小时 → 太粗，应拆分
- 目标：wiki 概念页总数控制在 **20-30 个**

### 2.3 合并执行流程

```
1. 列出合并候选组，展示给用户：
   "以下页面建议合并为 [[新概念名]]：
    - page-a.md（30行，1来源）— 保留核心定义
    - page-b.md（25行，2来源）— 保留第二段实操建议
    - page-c.md（18行，1来源）— 内容已被 page-b 覆盖，仅保留来源"

   ⏸️ 【检查点】等用户确认合并方案后再执行。

2. 确保 wiki/_archive/concepts/ 目录存在（不存在则创建）

3. 创建新的合并页面 wiki/concepts/new-merged-name.md：
   - 合并去重规则：
     a. 同一事实保留来源最多/最新的版本
     b. 不同角度的描述保留全部，用 H2/H3 分节
     c. 矛盾信息保留两者并标注 ⚠️
     d. Sources 合并所有子页面的来源（去重）
   - 合并后页面必须通过密度测试（3句话说不清 = 合格独立页）

4. 将旧的子页面移到 wiki/_archive/concepts/（保留完整内容，不删除）

5. 全局搜索并替换旧页面的 [[wikilinks]]：
   - 搜索范围：wiki/ 下所有 .md 文件
   - 将 [[old-page-name]] 替换为 [[new-merged-name]]
   - 如果替换后出现重复链接，去重

6. 更新 index.md
7. 记录 log.md
```

> ⚠️ **回滚机制**：如果合并后发现关键信息丢失，从 `_archive/` 恢复原始页面并撤销链接替换。

## 第三步：Insight 蒸馏与冻结

### 3.1 Insight 膨胀检测

当 insight 页面满足以下任一条件时，触发冻结：
- 超过 **100 行**
- **To Verify** 列表超过 10 条
- **Related** 超过 15 个链接
- 包含超过 3 周的"每日行动项"

### 3.2 冻结流程

```
1. 展示冻结计划给用户：
   "insight 页 codm-aigc-workflow.md 已达 {N}行，建议冻结：
    - 快照归档 → wiki/_archive/insights/codm-aigc-workflow-snapshot-YYYYMMDD.md
    - 精炼版保留核心框架（≤50行）+ 最强5条证据"

   ⏸️ 【检查点】等用户确认后再执行。

2. 确保 wiki/_archive/insights/ 目录存在（不存在则创建）

3. 将当前 insight 页面复制为带日期后缀的快照版本：
   codm-aigc-workflow.md → wiki/_archive/insights/codm-aigc-workflow-snapshot-YYYYMMDD.md

4. 创建新的精炼版 insight 页面 codm-aigc-workflow.md：
   - 精炼规则：
     a. 一句话核心洞察（≤30字）— 从全文中提炼最核心的一句
     b. 核心框架/方案（≤50行）— 保留结构和逻辑，删除重复论述
     c. 3-5 条最强支撑证据 — 按"证据强度"排序，只留 Top 5
     d. 1 条当前最重要的行动建议 — 替换所有旧行动项
     e. Related ≤10 个 — 按关联度排序取 Top 10
     f. To Verify ≤5 条 — 只保留可能影响决策的关键未验证项

5. 同时创建一篇"本周知识蒸馏"记录：
   wiki/insights/weekly-distill-YYYYMMDD.md
   记录本次蒸馏的变更摘要
```

### 3.3 Insight 页面硬限制

| 维度 | 限制 |
|------|------|
| 总行数 | ≤100 行 |
| Supporting Evidence | ≤5 条 |
| To Verify | ≤5 条 |
| 行动项 | ≤3 条（只保留当前最优先的） |
| Related | ≤10 个 |
| 不允许追加"每日行动项" | 行动项应替换不应追加 |

---

## 第四步：实体页信息衰减

### 4.1 实体页精简规则

每个实体页硬限制 **≤50 行**：

```markdown
# 实体名称

> 一句话定义（≤20字）

## Info 表（≤5行）

## 核心能力（≤5 条要点）

## CODM 怎么用（≤3 条）

## Related（≤10 个）

---
<details>
<summary>Timeline & Sources（折叠）</summary>
（时间线和详细来源放在这里）
</details>
```

### 4.2 Timeline 精简规则

- 只保留 **里程碑事件**，日常更新不入 timeline
- 同一事件的多次更新只保留最终状态
- 超过 30 天的非里程碑事件从 timeline 移除

### 4.3 异常处理

- **实体页已≤50行**：跳过，不做无意义修改
- **实体页无 Timeline**：跳过折叠区创建
- **精简后核心信息不足3条**：考虑合并到上级概念页，参照第二步合并流程

---

## 第五步：Related 区域精简

### 5.1 硬限制

每个页面的 Related 区域：**≤10 个链接**

### 5.2 超出处理

如果某页面当前 Related >10：
1. 保留 **最核心的 10 个**（直接相关的概念/实体）
2. 其余放入折叠区：

```markdown
## Related

- [[核心关联1]]
- ...（最多10个）

<details>
<summary>See Also</summary>

- [[次要关联1]]
- ...
</details>
```

---

## 第六步：更新系统文件

1. 更新 `index.md`：反映合并/删除/新建
2. 更新 `CODEBUDDY.md`：将 distill 操作加入 §3
3. 记录 `log.md`：本次蒸馏的完整变更清单

## 第七步：输出蒸馏报告

```
## Distill 蒸馏完成 — YYYY-MM-DD

### 变更摘要
- 概念页: 52 → N（合并 X 组，归档 Y 个）
- 实体页: 26 → N（精简 X 个）
- Insight: 冻结 X 个，新建 X 个
- Related 精简: X 个页面超限已修正

### 信息密度提升
- 平均页面行数: N → N
- 平均 Related 数: N → N
- Insight 页行数: N → N

### 知识库健康度
| 指标 | 蒸馏前 | 蒸馏后 |
|------|--------|--------|
| 概念页数 | X | X |
| 平均来源数/页 | X | X |
| 孤立页面 | X | X |
| Related均值 | X | X |
```

---

## 质量红线

1. **不丢信息**：合并和精简只改变组织方式，核心知识不丢失（归档到 `_archive/`）
2. **链接完整**：合并后所有 `[[wikilinks]]` 必须重定向到新页面
3. **index.md 同步**：所有变更反映到索引
4. **log.md 记录**：完整记录本次蒸馏操作
5. **密度提升**：蒸馏后平均页面信息密度必须提高（行数减少但来源数不减少）
